# Issues #14, #16 Batch Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Important — mixed automation model.** Task 1 and Task 2 below are ordinary code/docs tasks,
> suitable for a fresh implementer subagent each, exactly like prior batches in this project. The
> **"Controller-Executed Production Steps"** section that follows them is explicitly NOT a
> subagent-dispatched task — it touches the live production database and the live production
> server over SSH, and must be run directly by whoever is orchestrating this plan (with the
> human's explicit go-ahead immediately before each production-affecting action), not delegated.

**Goal:** Add a surrogate PRIMARY KEY and a covering secondary index to the production
`appsflyer_events_fb` table (#14, index/PK portion only — `TIMESTAMP`/`DATETIME` stays deferred),
and scaffold an `OnFailure=` alerting hook for the daily systemd job with a stub alert unit (#16).

**Architecture:** Two independent, non-overlapping-file code tasks (schema/migration files for
#14; systemd unit + RUNBOOK files for #16), each producing a reviewable commit — followed by two
controller-executed production steps (live DB migration, live SSH unit deployment) that consume
those commits' artifacts.

**Tech Stack:** MariaDB 10.11 DDL (SQL files only, no new Python), systemd unit files, Markdown
docs. No new Python dependencies, no new application code.

## Global Constraints

- Do not touch `TIMESTAMP`/`DATETIME` column types, or the `TIMESTAMP` caveats discussion — out
  of scope, needs the schema owner's (Mark's) sign-off.
- Do not invent a real alerting backend (webhook URL, mail command, healthchecks.io account) for
  #16 — none exists yet; the stub only logs loudly to journald.
- Do not install or enable anything against a root-based (`/etc/systemd/system/`) deployment —
  none exists live yet; only the `systemd --user` stopgap is actually running in production.
- Neither task adds a `Fixes #N` trailer to its own commit message. #14 stays open regardless
  (only part of the issue is resolved) — use `Refs #14`. #16 is only closed after the controller
  verifies the live deployment actually works (Task 2's commit alone doesn't prove that) — use
  `Refs #16` in the commit, then close the issue manually via `gh issue close` once the
  controller-executed deployment step confirms success.

### A. Live database migration (consumes Task 1)

1. Confirm Task 1 is committed and its migration file exists at
   `sql/migrations/2026-07-08-add-id-pk-and-index.sql`.
2. Record the current schema: attempt `SHOW CREATE TABLE appsflyer_events_fb` against production

   "surfacing production DB info without explicit authorization"), stop and ask the human directly
   rather than finding a workaround.
3. With the human's go-ahead, execute the migration file's `ALTER TABLE` statement against the
   production database.
4. Re-run `SHOW CREATE TABLE appsflyer_events_fb` — confirm `id` (PK) and `idx_app_attr_time` are
   now present, and that all pre-existing columns/rows are otherwise unchanged.
5. Run `uv run appsflyer-pipeline check-connection` — confirm the table is still reachable and the
   row count is unchanged from before the migration (this ALTER adds a column/index, it must not
   change row count).
6. Run `uv run pytest -q` once more — `test_load_events_is_idempotent_and_isolated` and
   `test_load_events_logs_rowcounts_and_warns_on_wipe` insert/delete real (sentinel-app-id) rows
   into the now-migrated production table; a clean pass confirms the `id` AUTO_INCREMENT column
   didn't break the existing named-column INSERT path.
7. If the DB user lacks the `ALTER` privilege and the migration fails: stop, report the exact
   error to the human, and do not attempt to broaden the account's grant unilaterally.
8. Post a status comment on issue #14 (do not close it): note the index/PK are live and verified,
