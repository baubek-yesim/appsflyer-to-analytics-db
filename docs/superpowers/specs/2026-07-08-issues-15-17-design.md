# Design: issues #15, #17 — 1M-row Pull API cap guard, docs/config polish

**Date:** 2026-07-08
**Issues:** [#15](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/15),
[#17](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/17)
**Status:** approved in brainstorming, ready for implementation planning

## Goals

1. **#15 — guard against the Pull API's undocumented-at-runtime 1M-row cap.** AppsFlyer's Pull
   API silently truncates raw-data exports beyond 1,000,000 rows with no error. `fetch_events`
   (`src/appsflyer_pipeline/appsflyer_client.py`) has no awareness of this: a chunk that hit the
   cap would load truncated data and report success. At current volumes (~1.3K rows/90 days)
   this can't happen — filed as low-priority hardening so a future orders-of-magnitude volume
   change fails loudly instead of silently truncating.
2. **#17 — four small non-behavioral cleanups found during a full repo review:**
   - `deploy/user-level/appsflyer-daily.service:4-5` — `Wants=`/`After=network-online.target`
     are dead config in a `systemd --user` manager (that target doesn't exist there); copied
     over from the root-based unit where it's meaningful.
   - `docs/design-spec.md:98` — says "see the backfill-window risk **above**" when the Risks &
     Failure Modes table is below, not above, that section.
   - Decorative `noqa` codes: `# noqa: S608` (loader.py ×3, test_loader_integration.py ×1) and
     `# noqa: RUF012` (config.py) suppress nothing because neither the `S` nor `RUF` rule
     groups are enabled in `pyproject.toml`'s `[tool.ruff.lint] select`. Re-verified during
     brainstorming with `ruff check --select RUF100 .`, which also surfaced **5 more**
     decorative `# noqa: BLE001` comments in `test_loader_integration.py` that the issue text
     didn't name — same root cause, same fix.
   - `README.md`'s Development section lists ruff/mypy/pytest but omits
     `uv run pre-commit run --all-files`, which `CLAUDE.md` calls out as the actual
     pre-push/CI gate.

## Non-goals

- Issues #14, #16 (schema/PK/index change, alerting) — tracked separately as "Batch B", each
  gated on a decision the user already made going in (do the additive index/PK now and defer
  the `TIMESTAMP`/`DATETIME` question to the schema owner; scaffold an `OnFailure=` hook with no
  real alerting backend yet).
- Enabling the `S` (flake8-bandit) or `RUF` rule groups in ruff. Tested during brainstorming:
  `ruff check --select S,RUF .` on the current tree returns **161 errors**, almost all `S101`
  ("use of assert detected") false positives across the pytest suite — normal, idiomatic pytest
  usage, not a real risk. Adopting either rule group wholesale is a separate, disproportionately
  larger effort than "these two/six noqa comments are decorative"; out of scope for this batch.
  `RUF100` (unused-noqa) is narrow enough to adopt on its own and directly fixes the reported
  problem, so it's in scope.
- Making `MAX_CHUNK_DAYS` operator-tunable (#15 calls this out as optional). YAGNI — nothing
  exercises it at current volume; add only if the row-cap guard ever actually fires.

## Design

### Part 1 — #15: 1M-row cap guard (`src/appsflyer_pipeline/appsflyer_client.py`)

In `fetch_events`, immediately after the CSV parses successfully (i.e. after the existing
`pl.exceptions.ComputeError` guard added by issue #11), check the resulting DataFrame's row
count:

```python
try:
    df = pl.read_csv(BytesIO(content), infer_schema_length=0)
except pl.exceptions.ComputeError as exc:
    raise AppsFlyerAPIError(
        f"AppsFlyer returned an unparseable CSV [{attribution_type}] for {app_id} "
        f"({from_date} to {to_date}): {exc}"
    ) from exc
if df.height >= 1_000_000:
    raise AppsFlyerAPIError(
        f"Report for {app_id} [{attribution_type}] {from_date}..{to_date} hit the "
        f"Pull API 1M-row cap — data is likely truncated; split the window into smaller chunks."
    )
return df
```

`AppsFlyerAPIError` is already isolated per-window in `_process_window` (`pipeline.py`) — no
other file changes needed. The threshold (`1_000_000`) matches AppsFlyer's own documented cap
exactly (`>=`, not `>`, since AppsFlyer's own bulletin describes the cap as inclusive at
1,000,000).

### Part 2 — #17: docs/config polish

**a) `deploy/user-level/appsflyer-daily.service`** — replace lines 4-5:
```ini
Wants=network-online.target
After=network-online.target
```
with a comment explaining the omission (matching this file's existing style of documenting
*why* a directive is absent, e.g. the `ProtectHome`/`ProtectSystem` comment block already in
this file):
```ini
# No Wants=/After=network-online.target here (unlike the root-based unit): that target does
# not exist in a systemd --user manager. The 05:00 OnCalendar fire time plus the timer's
# RandomizedDelaySec is the real guarantee the network is up by the time this runs.
```
The root-based unit (`deploy/appsflyer-daily.service`) is unchanged — `network-online.target`
is valid there.

**b) `docs/design-spec.md:98`** — change "see the backfill-window risk above" to "see the
backfill-window risk below".

**c) Decorative `noqa` cleanup + `RUF100`:**
- `pyproject.toml`: add `"RUF100"` to `[tool.ruff.lint] select` (alongside the existing `["E",
  "F", "I", "UP", "B", "SIM", "C4"]`).
- Remove all 10 now-flagged decorative comments (verified via `ruff check --select RUF100 .`
  during brainstorming):
  - `src/appsflyer_pipeline/loader.py:85,173,180` — `# noqa: S608` (×3)
  - `src/appsflyer_pipeline/config.py:47` — `# noqa: RUF012`
  - `tests/test_loader_integration.py:31,49,63,74,159` — `# noqa: BLE001` (×4 on `except
    Exception`) and `# noqa: S608` (×1, line 125)
- The underlying code these comments were annotating is unchanged — `_validate_identifier`
  still guards every one of these f-strings before they reach raw SQL, and the broad
  `except Exception` catches in `test_loader_integration.py` still exist for the same reason
  (skip gracefully in environments without a reachable DB); only the inert lint-suppression
  comments go.

**d) `README.md`** — add a line to the Development section:
```bash
uv run pre-commit run --all-files  # same checks CI gates on
```

## Tests

**`tests/test_appsflyer_client.py`** (Part 1): one new test, following the file's existing
`@respx.mock` convention. Constructing an actual 1M-row CSV body in a unit test is wasteful;
instead monkeypatch `appsflyer_client.pl.read_csv` to return a stub object exposing only
`.height = 1_000_000` (the one attribute `fetch_events` reads before returning), keeping the
test fast and avoiding any real large-data generation. Asserts `AppsFlyerAPIError` with a
message match on `"1M-row cap"`.

Part 2 is docs/config only — no test changes. `pre-commit` (whitespace/yaml checks) still
covers the touched files, and `pyproject.toml`'s new `RUF100` entry is verified by running
`uv run ruff check .` clean after the noqa removals (part of the task's own verification step,
not a new automated test).

CI gate: `pytest --cov-fail-under=98` branch coverage — the new `if df.height >= 1_000_000`
branch needs the covering test above.
