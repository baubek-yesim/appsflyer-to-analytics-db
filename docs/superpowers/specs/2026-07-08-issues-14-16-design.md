# Design: issues #14, #16 — target table PK/index migration, daily-run failure alerting scaffold

**Date:** 2026-07-08
**Issues:** [#14](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/14),
[#16](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/16)
**Status:** approved in brainstorming, ready for implementation planning

## Goals

1. **#14 — add a surrogate PRIMARY KEY and a covering secondary index to `appsflyer_events_fb`.**
   The table has neither today. Consequences already observed/reasoned about in the issue:
   `load_events`'s idempotency DELETE (`loader.py:172-176`, filtered on `app_id`,
   `attribution_type`, and an `event_time` range) full-scans the table on every call — 24+ scans
   per backfill, each one locking effectively every row it scans under InnoDB REPEATABLE READ, not
   just the window being replaced. Harmless at ~3.3K rows today; steadily worse as the table grows
   daily via the live daily timer.
   - **In scope for this batch (user decision):** add `id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY
     KEY` and `KEY idx_app_attr_time (app_id, attribution_type, event_time)`, applied to both the
     live production table (via a one-time migration, executed this session — DB credentials in
     `.env` already proven working via `check-connection`) and the schema template used for any
     future fresh install.
   - **Explicitly deferred (user decision):** the `TIMESTAMP` vs `DATETIME` question (2038 ceiling,
     session-timezone dependence, the `explicit_defaults_for_timestamp` caveat). The schema is
     Mark Malovichko's DDL — changing a column type needs his sign-off, not a unilateral change.
     Issue stays open with a status comment, same treatment as #8.
2. **#16 — scaffold an `OnFailure=` alerting hook for the daily systemd job**, so a failed run stops
   being silent-until-someone-runs-`journalctl`. **In scope (user decision):** wire the hook and a
   stub alert unit that logs a loud, greppable, high-priority journal entry — no real notification
   backend (Slack/Telegram/webhook/healthchecks.io) exists yet, so none is invented. The stub is
   built to be swapped for a real command later without touching the daily service unit again.

## Non-goals

- Issues #15, #17 — already closed (`77dde34`), tracked in the separate "Batch A" spec
  (`docs/superpowers/specs/2026-07-08-issues-15-17-design.md`).
- #14's `TIMESTAMP`/`DATETIME` reconsideration and the `PK`-vs-schema-owner sign-off it needs —
  explicitly out of scope, per above.
- Making the `id` PRIMARY KEY meaningful to any application code. Nothing in `loader.py` reads or
  writes `id` — it exists purely so InnoDB has an explicit clustering key instead of its hidden
  6-byte row id, and so future row-level tooling has something to reference. `_INSERT_COLUMNS`
  (`loader.py:131-149`) is unchanged; `id` is `AUTO_INCREMENT`, so every existing insert path
  continues to omit it exactly as it already omits any column it doesn't set.
- A real alerting backend for #16 (webhook, mail, healthchecks.io dead-man's switch). The issue
  documents both as options; neither is available yet. The dead-man's switch remains documented
  as a follow-up option because — unlike `OnFailure=`, which only fires when the service exits
  non-zero — it also catches "the timer silently stopped firing at all," a distinct failure mode
  `OnFailure=` structurally cannot detect.
- Rolling out the root-based unit files live. Only the `systemd --user` stopgap
  (`deploy/user-level/`) is actually installed on the production server today (per
  `docs/RUNBOOK.md` §14) — the root-based files (`deploy/appsflyer-daily.service` +
  its new alert unit) are updated in the repo for whenever the canonical deploy happens, but are
  not pushed to any server this batch (there is nowhere to push them to yet).

## Design

### Part 1 — #14: PK + index migration

**a) Schema template changes (future fresh installs)** — `sql/create_table.sql` and
`src/appsflyer_pipeline/loader.py`'s `_CREATE_TABLE_TEMPLATE` (the file header of
`create_table.sql` already mandates keeping the two in sync) both gain:
```sql
CREATE TABLE IF NOT EXISTS `appsflyer_events_fb` (
    `id`                    BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    `event_time`            TIMESTAMP      NOT NULL,
    ... (unchanged columns) ...
    `app_id`                VARCHAR(100)   NOT NULL,
    PRIMARY KEY (`id`),
    KEY `idx_app_attr_time` (`app_id`, `attribution_type`, `event_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```
`id` is the first column (conventional placement for a surrogate PK) — cost-free for a *fresh*
`CREATE TABLE`, unlike an `ALTER` on a populated table (see (b)).

**b) One-time migration for the existing production table** — new file
`sql/migrations/2026-07-08-add-id-pk-and-index.sql`:
```sql
-- One-time migration for issue #14: appsflyer_events_fb was originally created without a
-- PRIMARY KEY or any secondary index. This adds both to bring an already-provisioned table in
-- line with the schema in sql/create_table.sql (2026-07-08 onward).
--
-- NOT idempotent, NOT run automatically by the pipeline (create-table only ever does
-- `CREATE TABLE IF NOT EXISTS`, which does not retrofit an existing table) -- run this by hand,
-- once, against any environment that was provisioned before this date. A fresh environment
-- (a clean database, CI's mysql:8 service container) never needs this file: it gets the full
-- schema directly from sql/create_table.sql.
ALTER TABLE `appsflyer_events_fb`
    ADD COLUMN `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY FIRST,
    ADD KEY `idx_app_attr_time` (`app_id`, `attribution_type`, `event_time`);
```
This is executed directly against the production database as part of this batch (Task 1 of the
implementation plan), using the working credentials already in `.env`. `SHOW CREATE TABLE
appsflyer_events_fb` is checked before and after to confirm the change applied as expected. At
~3.3K rows this completes in well under a second regardless of `InnoDB`'s chosen algorithm for the
combined `ADD COLUMN ... FIRST` + `ADD KEY` (which, because of the column reorder, requires a
table rebuild rather than qualifying for `ALGORITHM=INSTANT` — irrelevant at this row count, but
noted since it's the reason `id` isn't appended at the end instead).

If the configured DB user turns out to lack the `ALTER` privilege (`docs/RUNBOOK.md`'s
prerequisites list only `SELECT`/`INSERT`/`DELETE`/`CREATE` for this account), the migration step
fails loudly and is escalated back rather than attempting to broaden the grant unilaterally.

**c) Documentation** — `docs/RUNBOOK.md` gets a short note (near the existing schema/table
references) that the live schema now includes `id` (PK) and `idx_app_attr_time`, and points at the
new `sql/migrations/` file for how that got there.

**d) Issue #14 stays open** with a status comment: (a)/(b) done and verified live; TIMESTAMP vs
DATETIME remains open pending Mark's input.

### Part 2 — #16: OnFailure alerting scaffold

**a) Wire the hook** — both `deploy/appsflyer-daily.service` and
`deploy/user-level/appsflyer-daily.service` gain, in `[Unit]`:
```ini
OnFailure=appsflyer-alert@%n.service
```
(`%n` expands to the full escaped unit name of the service that failed — here, always
`appsflyer-daily.service`, but written generically per systemd convention for `OnFailure=`.)

**b) Stub alert units** — two new template unit files, one per deploy variant (mirroring how
`appsflyer-daily.service` already has root/`user-level` twins):

`deploy/appsflyer-alert@.service` (root):
```ini
[Unit]
Description=Alert stub for a failed AppsFlyer unit: %i (BAF-2)
Documentation=https://yesimapp.atlassian.net/browse/BAF-2

[Service]
Type=oneshot
# STUB (issue #16): no real notification backend exists yet (no Slack/Telegram webhook, no
# healthchecks.io account). This only guarantees a failure is loud and greppable in journald --
# it does not page anyone. Replace ExecStart with a real `curl` to a webhook, a `mail` command,
# or similar once a backend is available; nothing else about this unit or appsflyer-daily.service
# needs to change when that happens. See docs/RUNBOOK.md's "Alerting on failure" section.
ExecStart=/usr/bin/systemd-cat -t appsflyer-alert -p crit /bin/echo "ALERT: %i failed -- run: journalctl -u %i --since today"
```

`deploy/user-level/appsflyer-alert@.service`: identical `[Unit]`/`[Service]` content (this stub
needs no privilege and has nothing to harden beyond what a one-line `echo` already implies), with
the description suffixed `[user-level stopgap]` to match this repo's existing naming convention
for the `--user` twin files.

**c) Install steps** — `docs/RUNBOOK.md` §7 (root install) and §14 (no-root stopgap, what's
actually live) both get the new alert unit added to their `install`/copy steps, followed by the
existing `daemon-reload` each section already runs.

**d) New "Alerting on failure" subsection under §10 (Monitoring)** documents:
- What `OnFailure=` does and its structural limit (only fires on a non-zero exit; cannot detect
  "the timer never fired at all").
- How to test the wiring without breaking the real job: `systemctl --user start
  appsflyer-alert@appsflyer-daily.service` (or `sudo systemctl start
  appsflyer-alert@appsflyer-daily.service` for the root variant), then `journalctl --user -u
  appsflyer-alert@appsflyer-daily.service` (drop `--user` for root) to see the stub's message.
- How to swap in a real backend later (edit `ExecStart` in the alert unit only).
- The dead-man's-switch alternative from the issue, documented as a follow-up option precisely
  because it catches the "timer never fired" gap `OnFailure=` cannot.

**e) Live deployment** — since only the `systemd --user` stopgap is actually installed on the
production server (`docs/RUNBOOK.md` §14), this batch deploys the updated
`appsflyer-daily.service` and the new `appsflyer-alert@.service` there via the SSH access already
on file, runs `systemctl --user daemon-reload`, and verifies via the test command in (d) — without
deliberately breaking the real daily job. The root-based unit files are updated in the repo only;
nothing is installed against a root-based deployment (none exists yet to install onto).

**f) Issue #16 is closed** once deployed and verified — the scope the user chose ("scaffold only,
nothing available to wire to yet") is fully satisfied; there's no dangling sub-question the way
#14 has one.

## Tests

Both parts are infrastructure/schema changes with no new application-level logic in
`src/appsflyer_pipeline/`, so neither introduces new Python unit tests:

- **Part 1:** `tests/test_loader_integration.py`'s existing `test_create_table_is_idempotent` and
  `test_load_events_is_idempotent_and_isolated` continue to exercise `create_table`/`load_events`
  against a real database (CI's `mysql:8` service, or a local `.env`) — updating
  `_CREATE_TABLE_TEMPLATE` means these tests now also implicitly verify the new template creates
  successfully (a fresh `mysql:8` container in CI has no pre-existing table, so `create_table`
  there exercises the *new* PK+index schema end-to-end, not the migration path). No test targets
  `sql/migrations/2026-07-08-add-id-pk-and-index.sql` directly — it's a one-time hand-run script,
  not application code.
- **Part 2:** unit files and RUNBOOK prose only; no Python code changes, hence no pytest coverage.
  Verification is the live `systemctl`/`journalctl` check described in Part 2(d)/(e), plus the
  project's existing `pre-commit`/`ruff format` checks (unit files and Markdown are still linted
  for whitespace/formatting).

CI gate: `pytest --cov-fail-under=98` — unaffected, since no new branches are introduced in
`src/appsflyer_pipeline/`.

## Delivery

- One branch, cut from `main`: `issues-14-16-batch-b`.
- Given the production-affecting steps (a live `ALTER TABLE`, a live systemd unit install over
  SSH), each of those specific actions gets its own explicit go-ahead at execution time, in
  addition to the plan approval — consistent with how #7's data purge and #19's live hotfix were
  handled.
- Merge only on the user's explicit authorization, per established workflow.
- Commit messages: both `Refs #14` and `Refs #16`, not `Fixes` — neither task's own commit is
  what decides closure. #14 stays open by design (only part of the issue is resolved). #16 is
  only closed by the controller once the live deployment step actually verifies it works, since a
  clean commit alone doesn't prove that; using `Fixes #16` on the commit would risk an auto-close
  on push before that verification ever ran. `Refs #N` links each commit to its issue's timeline
  without closing it either way.
