# Design: batch fix for #27 (ValidationError secret leak), #28 (retention-floor anchor), #29 (empty scalar config)

**Date:** 2026-07-09
**Issues:** [#27](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/27),
[#28](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/28),
[#29](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/29) — three P2s from the
2026-07-09 full-repo review, batched (independent files, one theme: operator-input robustness).
**Status:** approved in brainstorming, ready for implementation planning

## Goals

1. **#27:** a config error must never print secret material. Today `FAILED: {ValidationError}`
   embeds pydantic's `input_value=<collected env dict>` repr — reproduced leaking a fake
   DB-password tail into stderr/journald when the EnvironmentFile is truncated after
   `DB_PASSWORD` (the exact scenario #9 called realistic).
2. **#28:** the backfill retention-floor warning must anchor to **today**, not `--end-date`
   (`default_start = end - 89d` silently passes fully-beyond-retention windows when an explicit
   past end date is given), and `daily --date` must gain the same warning (it has none).
3. **#29:** empty scalar env values must fail startup loudly. `APPSFLYER_MEDIA_SOURCE=""` is
   accepted today (verified) and converts every window into a filter-wipe that exits 0 — the
   same truncated-EnvironmentFile class #9 fixed for the two CSV list fields only.

## Non-goals

- **No `SecretStr` migration (#27).** The leak is pydantic's *pre-validation* input dict, which
  `SecretStr` cannot mask (stated in the issue); the render fix closes the leak fully, and
  `SecretStr` would touch every credential call site for no additional protection here.
- **No #45 material in #28** (explicit user scoping decision: "minimal as filed"). The warning
  keeps using the nominal `MAX_RETENTION_DAYS = 90`; no configurable availability floor, no
  populated-window wipe guard — #45 stays open and untouched. #28's docstring gains a one-line
  cross-reference to #45 (words only, zero behavior).
- **No clamping (#28).** Warn-and-proceed for beyond-floor requests stays — it is deliberate
  evidence-gathering behavior (design-spec Interfaces; RUNBOOK §9 relies on it).
- **`db_password` keeps accepting empty (#29)** — CI's mysql:8 service uses an empty root
  password (`.github/workflows/ci.yml:48`); pinned by a test.
- No changes to `PipelineError` rendering (SQLAlchemy masks DSN passwords in its messages —
  checked during the review), no loader/transform/client changes.

## Context

- `src/appsflyer_pipeline/cli.py:96-105` — `_get_settings_or_exit` prints `f"FAILED: {exc}"`;
  the same raw rendering sits in `backfill` (`cli.py:143-147`) and `daily` (`cli.py:172-177`)
  `except (PipelineError, ValidationError)` handlers, which catch the `get_settings()`
  ValidationError raised inside `run_backfill`/`run_daily`.
- `src/appsflyer_pipeline/pipeline.py:251-266` — `run_backfill` computes
  `default_start = end - timedelta(days=MAX_RETENTION_DAYS - 1)` and warns only when
  `start < default_start`; with the default `end = yesterday` this coincides with the true
  floor, which is why `tests/test_pipeline.py::test_run_backfill_warns_but_does_not_clamp_early_start`
  passes while the explicit-past-end-date case silently under-warns.
  `run_daily(date=...)` (`pipeline.py:284-285`) has no floor check.
- `src/appsflyer_pipeline/config.py:30-46` — scalar fields are plain `str`; only the two
  `CsvList` fields carry `min_length=1` (issue #9).
- `src/appsflyer_pipeline/cli.py:129-131` — `--start-date` help reads "defaults to 90 days
  before yesterday", wrong when `--end-date` is given.

## Design

### Fix 1 — #29: reject empty scalar env values (`src/appsflyer_pipeline/config.py`)

`Field(min_length=1)` on exactly six fields: `db_host`, `db_user`, `db_name`, `db_table`,
`appsflyer_api_token`, `appsflyer_media_source`. `db_password` and `db_port` unchanged.
A comment mirrors the #9 rationale (truncated hand-edited EnvironmentFile line must fail startup
loudly; an empty media source is a filter-wipe at exit 0 — refs #29/#9).

Ordering note: this fix lands first so #27's tests can use an empty scalar as one of their
ValidationError triggers.

### Fix 2 — #27: leak-free ValidationError rendering (`src/appsflyer_pipeline/cli.py`)

New module-level helper:

```python
def _format_validation_error(exc: ValidationError) -> str:
    """Render a config error without echoing collected env values (issue #27):
    pydantic's default str() embeds input_value=<the collected settings dict>,
    which leaks DB-password/API-token material into stderr and journald when
    the EnvironmentFile is truncated (#9's exact scenario).
    """
    problems = "; ".join(
        f"{'.'.join(str(p) for p in err['loc'])}: {err['msg']}"
        for err in exc.errors(include_url=False, include_input=False)
    )
    return f"invalid configuration: {problems}"
```

Used in all three ValidationError render sites: `_get_settings_or_exit` and the
`backfill`/`daily` except handlers print `FAILED: {_format_validation_error(exc)}` for
ValidationError while keeping `FAILED: {exc}` for `PipelineError`. Example output:

```
FAILED: invalid configuration: db_name: Field required; db_table: Field required
```

### Fix 3 — #28: today-anchored floor warning (`src/appsflyer_pipeline/pipeline.py` + one help string)

In `run_backfill`:

```python
retention_floor = _today() - datetime.timedelta(days=MAX_RETENTION_DAYS)
...
if start < retention_floor:
    logger.warning(... same message shape, printing retention_floor ...)
```

`default_start` (the *window default*) stays end-anchored and is otherwise untouched; only the
warning predicate and the date it prints change. `run_daily` gains the mirror check on the
explicit-date path: when `date < retention_floor`, log the same warning before `_run_window`
(warn-and-proceed, identical philosophy). Docstrings note that the nominal 90-day floor is the
API's documented/HTTP-400 boundary and that the *silent-empty* boundary is shorter — see #45
(cross-reference only).

Deliberate message-text update while editing the warning: the current text ends with
"Proceeding anyway; this is expected until BAF-2's backfill-window conflict is resolved with
the stakeholder" — that conflict **was resolved** on 2026-07-07/08 (rolling window accepted,
history accumulated in our DB; BAF-2 comments 62493/62581). The rewritten warning keeps the
"Proceeding anyway" clause but replaces the stale rationale with the current one: requests
before the floor may return empty data or an error (per RUNBOOK §9 / #45).

`cli.py` `--start-date` help becomes: `ISO date (YYYY-MM-DD); defaults to a 90-day window
ending at --end-date (or yesterday).`

## Tests

- `tests/test_config.py` (#29): parametrized — each of the six fields rejects `""` and `"   "`
  with `ValidationError`; one test pins that `DB_PASSWORD=""` is still **accepted**.
- `tests/test_cli.py` (#27): sentinel secret never printed — set env with
  `DB_PASSWORD=<sentinel>` and a missing required field, invoke `check-connection`: exit 1,
  output contains `FAILED: invalid configuration:` and the missing field name, and does **not**
  contain the sentinel. A second test drives the `backfill` handler path (ValidationError raised
  from `run_backfill`'s `get_settings()`) with an empty `APPSFLYER_MEDIA_SOURCE` (#29's
  validator as the trigger) and asserts the same clean rendering + no sentinel.
- `tests/test_pipeline.py` (#28): (a) the filed blind spot — explicit past `--end-date` window
  fully beyond the floor → warning fires (fixed `_today()`, caplog on
  `appsflyer_pipeline.pipeline`); (b) `run_daily(date=<old>)` → warning fires; (c) a recent
  window → no warning; existing floor tests unchanged.
- All behavior verified through the existing seams (`pipeline._today` monkeypatch, `caplog`,
  `CliRunner`) — no new test infrastructure.
