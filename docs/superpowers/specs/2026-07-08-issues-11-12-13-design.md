# Design: issues #11, #12, #13 — per-window CSV isolation, CLI exception correctness, RUNBOOK sudo fix

**Date:** 2026-07-08
**Issues:** [#11](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/11),
[#12](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/12),
[#13](https://github.com/baubek-yesim/appsflyer-to-analytics-db/issues/13)
**Status:** approved in brainstorming, ready for implementation planning

## Goals

1. **#11 — isolate malformed-CSV failures per window.** `fetch_events` lets an uncaught
   `polars.exceptions.ComputeError` (ragged row, unclosed quote, an HTML error page delivered
   with HTTP 200) propagate out of `pl.read_csv`. `_process_window` only isolates
   `(AppsFlyerAPIError, TransformError, PipelineError)`, so a `ComputeError` kills the entire
   run — including windows that hadn't been processed yet — instead of failing just the one
   bad window, which is how every other upstream-data problem (schema drift, bad timestamps,
   bad revenue) is already treated.
2. **#12 — fix CLI exception-handling correctness.** Three related problems in
   `src/appsflyer_pipeline/cli.py`: (a) a docstring falsely claims pydantic v2's
   `ValidationError` doesn't subclass `ValueError` (it does); (b) `backfill`/`daily`'s
   `except (PipelineError, ValueError)` therefore catches `ValidationError` only by
   accident, and also swallows any unrelated `ValueError` a real bug might raise deep in a
   run, contradicting the project's "unexpected bugs must crash loudly" philosophy
   (`pipeline.py:118-124`); (c) `check-connection`/`create-table` call `get_settings()`
   outside any `try`, so the *same* config mistake that `backfill`/`daily` render as a clean
   `FAILED:` line instead dumps a full traceback for these two commands.
3. **#13 — fix the RUNBOOK's broken sudo invocation.** §4 and §13 use
   `sudo -u appsflyer --login bash -c '...'` against a service account created with
   `--shell /usr/sbin/nologin` (§2). `sudo --login` execs the target user's *login shell from
   `/etc/passwd`* — for this account, that's `nologin`, which ignores its arguments and exits 1
   without running anything. Never hit live because only the §14 no-root stopgap has actually
   been executed; a landmine for the future root-based migration.

## Design

### Part 1 — #11: `fetch_events` isolates a malformed CSV (`src/appsflyer_pipeline/appsflyer_client.py`)

Wrap the parse call:

```python
if not content.strip():
    return pl.DataFrame()
# infer_schema_length=0 forces every column to Utf8: chunks are read
# independently and later pl.concat'ed, so dtype inference (e.g. an
# all-null column guessed as Int64 in one chunk, Utf8 in another) must
# not be allowed to diverge between them. transform.py applies real types.
return pl.read_csv(BytesIO(content), infer_schema_length=0)
```

becomes:

```python
if not content.strip():
    return pl.DataFrame()
# infer_schema_length=0 forces every column to Utf8: chunks are read
# independently and later pl.concat'ed, so dtype inference (e.g. an
# all-null column guessed as Int64 in one chunk, Utf8 in another) must
# not be allowed to diverge between them. transform.py applies real types.
try:
    return pl.read_csv(BytesIO(content), infer_schema_length=0)
except pl.exceptions.ComputeError as exc:
    raise AppsFlyerAPIError(
        f"AppsFlyer returned an unparseable CSV [{attribution_type}] for {app_id} "
        f"({from_date} to {to_date}): {exc}"
    ) from exc
```

`AppsFlyerAPIError` is already isolated per-window in `_process_window` (`pipeline.py:165`) —
no other file changes. Verified live: a ragged-row body
(`"a,b,c\n1,2,3,4,5\n"` — header declares 3 columns, data row has 5) reliably raises
`polars.exceptions.ComputeError` with `infer_schema_length=0` ("found more fields than defined
in 'Schema'").

### Part 2 — #12: CLI exception-handling correctness

**`src/appsflyer_pipeline/pipeline.py`** — `run_backfill`'s inverted-range check changes from a
bare `ValueError` to the project's own error type (`PipelineError` is already imported in this
file):

```python
if start > end:
    raise ValueError(f"start {start} is after end {end}")
```
→
```python
if start > end:
    raise PipelineError(f"start {start} is after end {end}")
```

**`src/appsflyer_pipeline/cli.py`:**

- Fix `_parse_optional_date`'s docstring (currently claims `ValidationError` "does NOT
  subclass ValueError" — false for pydantic v2, verified: `issubclass(ValidationError,
  ValueError)` is `True`).
- Add `from pydantic import ValidationError` and extend the config import to
  `from appsflyer_pipeline.config import Settings, get_settings`.
- Add a small private helper (matching this file's existing style of small helpers like
  `_print_summary`):
  ```python
  def _get_settings_or_exit() -> Settings:
      try:
          return get_settings()
      except ValidationError as exc:
          typer.echo(f"FAILED: {exc}", err=True)
          raise typer.Exit(code=1) from exc
  ```
- `check_connection_command`/`create_table_command`: replace `settings = get_settings()` with
  `settings = _get_settings_or_exit()` — gives both the same clean-message behavior
  `backfill`/`daily` already have (currently they crash with a full traceback on bad config;
  after this they print `FAILED: ...` + exit 1, same as the other two commands).
- `backfill`/`daily`: change `except (PipelineError, ValueError)` →
  `except (PipelineError, ValidationError)` — explicit intent (both are now real exception
  types this code path can actually raise: `PipelineError` from the `start > end` check or
  DB-layer failures, `ValidationError` from `get_settings()` inside `run_backfill`/`run_daily`)
  instead of accidentally relying on `ValidationError` being a `ValueError` subclass, and no
  longer swallows an unrelated `ValueError` from a real bug elsewhere in the run.

Net effect: all four CLI commands now handle a bad/missing env var identically (clean
`FAILED:` message, exit 1), and a genuine unexpected bug (any exception type other than
`PipelineError`/`ValidationError`) still crashes loudly with a full traceback everywhere,
matching the project's stated philosophy.

### Part 3 — #13: RUNBOOK sudo fix (`docs/RUNBOOK.md`)

Both occurrences (§4 line 76, §13 line 287) change:
```bash
sudo -u appsflyer --login bash -c '
```
to:
```bash
sudo -u appsflyer env HOME=/opt/appsflyer bash -c '
```
Drops `--login` (which execs the account's `nologin` shell instead of running the given
command) while keeping the one thing it was providing — `HOME`, needed for uv's cache dir
(`~/.cache/uv`). Docs-only; no code or test changes, still runs through
`pre-commit` (whitespace linting applies to `.md` files too).

## Tests

**`tests/test_appsflyer_client.py`** (Part 1): one new test, following the file's existing
`@respx.mock` + `_url()` convention (modeled on
`test_fetch_events_raises_on_client_error_without_retry`):
- A ragged-row CSV body returned as HTTP 200 → `fetch_events` raises `AppsFlyerAPIError`
  (verifies the isolation is at the `AppsFlyerAPIError` boundary, which `_process_window`
  already isolates — no `pipeline.py` test needed since no new logic lives there).

**`tests/test_pipeline.py`** (Part 2): update the existing
`test_run_backfill_rejects_start_after_end` (currently `pytest.raises(ValueError, ...)`) to
`pytest.raises(PipelineError, match="after")` — `PipelineError` is already imported in this
file.

**`tests/test_cli.py`** (Part 2):
- Update `test_backfill_start_after_end_reports_failure`'s docstring (currently says
  "run_backfill() itself raises ValueError(...)" — now `PipelineError`); its assertions
  (`exit_code == 1`, `"FAILED" in result.output`) are unchanged since `PipelineError` is still
  caught the same way.
- Two new tests, matching the file's existing `_set_cli_env`/`get_settings.cache_clear()`
  convention (modeled on `test_check_connection_reports_failure_for_unreachable_db`) — verified
  live that an empty `APPSFLYER_APP_IDS` currently produces `exit_code=1`, `output=''`,
  `exception=<ValidationError>` (an uncaught crash, just one Click happens to also report as
  exit code 1) and will instead produce `"FAILED" in result.output` once fixed:
  - `check-connection` with `APPSFLYER_APP_IDS=""` → `exit_code == 1`, `"FAILED" in
    result.output`.
  - `create-table` with `APPSFLYER_APP_IDS=""` → same assertions.

CI gate: `pytest --cov-fail-under=98` branch coverage — every new branch needs a covering test;
all of the above are exercised by the tests listed.
